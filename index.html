1.2 Step1：开始计提

1 校验开关

检查计提开关：programmedSwitch=true 且 programmedConfigSwitch=false 才能操作。

配置来源：配置中心 tab「计提开关配置」

2 查配置

按用户选择的「计提人」查 config_miads 表（tab「miads计提配置」）。

获取字段说明
dspLevel1 / dspLevel2管辖的一级/二级DSP
accountId管辖的账户ID
accrualNo计提编号
am客户经理
accrualBy计提人
settlementCurrency结算币种
3 查原始消耗数据 + 聚合

数据源：accrual_automation（实体类 AccrualMiads）

按筛选条件查询，7维聚合：

GROUP BY dsp_level1, dsp_level2, package_name, region, aduser_account, agency_account, aduser_account_type SELECT ..., SUM(dsp_fee), SUM(dsp_active)
4 计算预计收入
// 核心公式 exDspFee = dspFee / accrualPeriod × dayByMonth // accrualPeriod：计提周期天数（tab「计提周期&合并推送」） // dayByMonth：目标月份实际天数（如5月=31天） // 含义：用已知天数的日均收入，推算整月的预计收入
5 绑定业务字段

通过 tab「miads计提配置」 匹配，设置 accrualNo、am、accrualBy、accountId。

⚠️
匹配规则：
• 效果类 DSP → 按 accountId 匹配
• online/offline 类 DSP → 按 dspLevel2 匹配
• 同一 accountId/dspLevel2 有多条配置时，取最后一条（遍历不 break，后面的覆盖前面的）
6 聚合到 9 维度
GROUP BY dspLevel1, dspLevel2, accrualNo, region, packageName, am, accrualBy, accountId, aduserAccountType

分别 SUM：dspActive、dspFee、exDspActive、exDspFee

产出：数据列表 + redisKey（传给后续步骤）

1.3 Step2：媒体替换 + 国家替换

💡
关键点：Step2 不依赖 Step1 的聚合结果，而是重新查 accrual_automation 原始表，用不同的维度重新聚合。Step1 的结果存 Redis 给 Step3 用。
① SQL 层：桌面媒体替换 + 账户合并

一条 SQL 两个 CASE WHEN 同时完成：

CASE WHEN 1：媒体名替换

配置来源：config_tag_media（tab「媒体收入拆分配置」）

字段说明
tag_id（广告位）广告位ID
media_type（替换前媒体）原始媒体名
new_media_type（替换后媒体）替换后的媒体名
CASE WHEN media_type='MSA' AND tag_id='1.386.4.101' THEN 'GLOBAL_368' ELSE media_type END AS new_media_type
⚠️
匹配条件：media_type + tag_id 同时命中才替换。
未命中则保留原始媒体名（ELSE 分支）。
CASE WHEN 短路逻辑：从上到下逐条判断，命中第一条就停，后面的同条件配置不生效。
代码从后往前遍历写入 CASE WHEN，所以配置表里排在后面的记录优先级更高。

CASE WHEN 2：账户ID合并

CASE WHEN aduser_account_type = '广告主' THEN aduser_account ELSE agency_account END AS account_id

根据账户性质二选一，不需要查配置文件。

SQL 聚合维度

GROUP BY dsp_level1, dsp_level2, region, package_name, new_media_type, account_id SELECT ..., SUM(dsp_fee)
❌
相比原始表丢失的字段：tagId（用掉了）、aduserAccount / agencyAccount（合并了）、aduserAccountType、dspActive（不聚合）
② Java 层：媒体拆分 + 媒体名称替换

SQL 结果转为 AccrualMiadsUp 对象后，new_media_type 赋值给 mediaType 字段。

2a. 媒体拆分（handleMedias 方法，从后往前遍历配置）

配置来源：config_region_media（tab「媒体收入分成」，有 dividedMedia + dividedRatio 的记录）

字段说明
beforeReplace（原始媒体名称）要拆分的媒体
dividedMedia（收入拆分到媒体）拆分出的子媒体名
dividedRatio（分成系数）拆分比例
💡
拆分逻辑：每条记录的 mediaType 匹配 beforeReplace 时，按 dividedRatio 拆分金额。
每次拆分基于当前剩余金额，不是原始金额。拆分完原记录丢弃。
// 拆分示例：MSA, dspFee=6000, exDspFee=12400 // 配置：MSA → GLOBAL_368 (0.24), MSA → GLOBAL_BROWSER (0.05) 第1轮 GLOBAL_368：dspFee = 6000 × 0.24 = 1440，剩余 4560 第2轮 GLOBAL_BROWSER：dspFee = 4560 × 0.05 = 228，剩余 4332 → 丢弃原 MSA 主记录，保留 2 条子记录

2b. 纯媒体名称替换

配置来源：config_region_media（tab「国家媒体替换」中 replaceType=媒体、无拆分字段的记录）

遍历所有记录（含刚拆分出的子记录），匹配 beforeReplace → 设置 newMedia = afterReplace。

③ Java 层：国家替换

配置来源：config_region_media（tab「国家媒体替换」中 replaceType=国家 的记录）

按 beforeReplace → afterReplace 映射替换国家，未匹配的归「其他」。

💡
执行顺序：① 媒体拆分 → ② 媒体名称替换 → ③ 国家替换。先处理完所有媒体相关操作，再处理国家替换。
→ Step2 输出
字段说明来源
dspLevel1 / dspLevel2 / region / packageName基础维度原始表
accountId账户IDCASE WHEN 合并
accrualNo / am / accrualBy业务字段Step1 绑定（Redis）
dspFee / exDspFee实际收入 / 预计收入拆分后可能变小
newMedia新媒体名SQL替换 + Java拆分/改名
newRegion新国家国家替换
oriExDspFee原始预计收入拆分前保存，供比较
upFlag / upReasons是否修改 / 修改原因用户操作

1.4 Step3：最终保存

1 获取 Step2 结果 + 应用用户修改

从 Redis 获取 Step2 结果（redisKey），缓存失效则重算 Step2。

应用用户在 Step2 页面的修改：通过 9 字段匹配（dspLevel1, accrualNo, dspLevel2, newRegion, newMedia, packageName, oriExDspFee, am, accrualBy）找到对应记录，覆盖 exDspFee 和 upReasons。

2 汇总到最终保存维度
// GROUP BY 维度（6维，比 Step2 少了 dspLevel2 / packageName / am） GROUP BY dspLevel1, accrualNo, newRegion, newMedia, accrualBy, accountId // 分别 SUM oriExDspFee 和 exDspFee
⚠️
维度变粗：Step2 是 9 维度（含 dspLevel2、packageName、am），Step3 汇总到 6 维度，这些字段被聚合掉了。
3 过滤无效数据

• 去掉 oriExDspFee ≤ 0 的记录

• 去掉 newMedia = "msa" 的记录

4 设置结算币种

从 tab「miads计提配置」 按 [dspLevel1 + accrualNo + accrualBy] 匹配，设置 currency（结算币种）。

5 汇率转换

配置来源：exchange_rate（tab「汇率配置」）

dspFee（结算币种金额）= exDspFee（美元预计收入）× exchangeRate
6 月环比计算

查 accrual_miads_automation 上月同维度记录，匹配维度：[dspLevel1 + accrualNo + newRegion + newMedia]

compareResult（绝对值）= 本月dspFee - 上月dspFee monthRatio（月环比%）= (本月dspFee - 上月dspFee) / 上月dspFee × 100%
⚠️
注意：月环比基于 dspFee（汇率转换后、返点前的值）计算。代理返点在月环比之后才算，所以月环比和返点后的值可能对不上。
7 代理返点计算

配置来源：config_miads_proxy_return_rate（tab「返点比例配置」）

按 accountId 匹配返点率：

returnConsume（返点后消耗）= dspFee - (rate × dspFee)

同时设置 accrualType = "1"（预估收入）。

8 保存到数据库 + 推送 CRM

当 save=true 时执行：

操作表名说明
删除旧数据accrual_miads_automation按 [dspLevel1, accrualNo, newRegion, newMedia] 匹配，先删后插
保存最终结果accrual_miads_automation6维度汇总数据
保存明细数据accrual_miads_upStep1 + Step2 的明细（用于计提记录查询）
推送 CRM—仅 dspLevel1="online" 的才推
✅
Gap 检测（导出时）：合同ID维度差异 ≥ $30,000 或 合同ID×国家×媒体维度差异 ≥ $5,000 → 标注「gap较大，需给出调整原因」

预装 二、预装计提完整流程

API：POST /accrualPreinstall/beginAccrual → POST /accrualPreinstall/preinstallStep2 → POST /accrualPreinstall/preinstallSave

⚠️
与 miads 的核心差异：
• 数据源完全不同：不从 accrual_automation 取数，从多个表拼接（三方表、客户上传表、SuperSet、MIUIBI）
• 收入计算方式不同：不是「实际消耗推算全月」，而是「单价 × 预计激活」
• 按结算方式（CPP/CPA/CPE/CPS）走 4 个完全不同的分支
• 只有国家替换，没有媒体替换和媒体拆分

2.2 Step1：开始计提（四种结算方式）

1 校验开关

检查：preinstallSwitch=true 且 preinstallConfigSwitch=false

与 miads 独立的开关配置。

2 查配置

按计提人查 config_preinstall（tab「预装计提配置」）

字段说明
accrualNo合同ID
customerAbbreviation客户简称
country国家
packageName包名
unitPrice单价
settlementModes结算方式（CPP/CPA/CPE/CPS）
preinstallMedia预装媒体
settlementCurrency结算币种
3 按结算方式分 4 个分支计算

CPP（Cost Per Preinstall）—— 出货量

项目说明
数据源accrual_automation_superset（SuperSet 出货量数据）
预装媒体硬编码「海外预装」
激活数activate = 出货量 × 100
预计激活exDspActive = activate / accrualPeriod × dayByMonth
收入dspFee = unitPrice（配置表固定） × exDspActive

CPA（Cost Per Activation）—— 三方激活

项目说明
数据源1accrual_automation_install_type（三方激活数据）
数据源2config_customer_data（客户上传本月数据）
数据源3config_customer_data（客户上传上月数据）+ accrual_automation_miuibi（MIUIBI 激活数据）
激活数三方/客户数据 × 100
预计激活exDspActive = activate / accrualPeriod × dayByMonth
收入dspFee = unitPrice（配置表固定） × exDspActive
⭐
CPA 特殊点：数据源3用 MIUIBI 激活比例推算本月激活：本月激活 = 上月客户账单激活 / 上月MIUIBI激活 × 本月MIUIBI激活。三个数据源按 [accrualNo + customerAbbreviation + country + packageName + preinstallMedia + accrualBy] 合并累加。

CPE（Cost Per Engagement）—— 动态单价

项目说明
数据源config_customer_data（上月）+ accrual_automation_miuibi（MIUIBI 激活）
单价动态计算：上月结算金额 / 上月MIUIBI激活数
激活数MIUIBI本月激活 × 100
预计激活exDspActive = activate / accrualPeriod × dayByMonth
收入dspFee = 动态单价 × exDspActive
💡
支持 GLOBAL 维度：不区分国家时，用全球汇总数据计算单价和激活数。

CPS（Cost Per Sale）—— DAU

项目说明
数据源config_customer_data（上月）+ accrual_automation_miuibi（MIUIBI DAU）
单价动态计算：上月结算金额 / (上月总DAU / 上月天数)
激活数本月DAU / 计提周期 × 100（实际是DAU）
预计激活exDspActive = activate（不做比例换算）
收入dspFee = 动态单价 × exDspActive
⚠️
CPS 与其他三种的关键区别：预计激活不再做 /accrualPeriod × dayByMonth 的比例换算，因为 CPS 的激活本身就是日均 DAU/计提周期的结果。
4 Buffer Time 计算

从 b_collection_contract（收款合同表）获取合同结束日期和 buffer time 配置：

bufferTime含义
0不加
1+30天
2+90天
3+180天
4+3个月
5+6个月
6+12个月
7+自定义天数

2.3 Step2：国家替换

1 重新执行 Step1 获取数据
💡
与 miads 类似，Step2 不依赖 Step1 的结果，而是重新调用 beginAccrual() 获取第一步数据。
2 国家替换

配置来源：config_region_media（tab「国家媒体替换」，replaceType=国家）

按 beforeReplace → afterReplace 映射，未匹配归「其他」。

⚠️
与 miads 的关键差异：预装 Step2 只有国家替换，没有媒体替换和媒体拆分。预装的媒体名（preinstallMedia）在 Step1 已确定，Step2 不再处理。
3 按合同维度聚合汇总
GROUP BY accrualNo, customerAbbreviation, newCountry, settlementModes, settlementCurrency, preinstallMedia, accrualBy

2.4 Step3：最终保存

1 获取 Step2 结果 + 应用用户修改

从 Redis 获取 Step2 结果，通过多字段匹配（accrualNo, customerAbbreviation, preinstallMedia, settlementModes, settlementCurrency, newCountry）更新用户修改。

2 汇率转换

配置来源：tab「汇率配置」

dspFee（结算币种金额）= exDspFee（美元）× exchangeRate
3 过滤 + 月环比 + 保存

• 过滤：oriExDspFee ≤ 0 的记录、预装媒体为 "msa" 的记录

• 月环比：匹配维度 [accrualNo + customerAbbreviation + newCountry + preinstallMedia]

• 计提类型默认设为 "1"（预估收入）

• 保存到 accrual_preinstall_automation（先删后插）

• 同时保存明细到 accrual_preinstall_up

• 所有记录都推送 CRM（与 miads 不同，miads 仅 online 推送）

• 推送时「海外云端预装」→「开机引导-PAI」

优化 4.1 流程简化：三步合一

! 现状问题

Step1 → Step2 → Step3，三次后端计算，用户操作四次点击。

⚠️
Step1 的计算结果（exDspFee 等数值）不被 Step2 直接使用（Step2 重新查库重算）。
但 Step1 的用户修改会传递到后续步骤：用户在 Step1 页面修改预计收入后，修改值通过 updateAm 参数传给 Step2，Step2 按比例重算媒体维度的预计收入，最终影响 Step3 的结果。
所以 Step1 不能简单删除，需要合并到新的流程中。
// Step1 用户修改传递链路 Step1 用户修改 exDspFee → 传给 Step2（updateAm 参数） → Step2 handUpdateData() 按比例重算媒体维度 exDspFee → 存入 Redis（step2Data） → Step3 取 step2Data 汇总 → 算汇率/返点/月环比 → 最终结果（受 Step1 修改影响）

💡
关键变化：
• 原 Step1（展示+修改）+ Step2（替换+计算）+ Step3（保存）合并为一步
• 用户在最后才调整预计收入，此时已经看到所有替换和计算后的完整结果

后端只保留两个接口计算（一次完整计算）+ 保存（保存+推CRM）

4.2 处理阶段重划：先变换，后计算

💡
核心原则：把「数据变换」和「数据计算」彻底分开。变换阶段只改字段名和记录条数，计算阶段一次性算完所有衍生值。
阶段做什么特点
阶段一：数据变换查原始数据 → 桌面媒体替换 → 账户合并 → 媒体拆分 → 媒体改名 → 国家替换只改字段名和记录条数，除拆分外不碰金额
阶段二：数据聚合按最终维度 [dspLevel1, accrualNo, newRegion, newMedia, accrualBy, accountId] 汇总确定最终维度
阶段三：数据计算预计收入 → 汇率转换 → 返点 → 月环比每个值只算一次
阶段四：用户调整（可选）用户看完最终结果后修改预计收入 → 前端按媒体占比自动分配 → 实时重算衍生值在最后一步调整，前端处理，不需要后端重跑
阶段五：持久化保存数据库 + 推CRM用户确认后执行
✅
好处：预计收入只算一次。变换和计算解耦，排查问题看对应阶段就行。用户在最后才调整，能看到完整结果再决定是否修改。修改在前端处理，不需要后端重跑替换拆分。

4.3 返点展示优化

! 现状问题

只存 returnConsume（返点后消耗）和 rate（返点率），用户看不到返点金额。

月环比基于返点前的 dspFee 计算，但用户看到的是返点后的值，两者对不上。

→ 优化方案
字段说明公式
exDspFee预计收入（美元）dspFee / 周期 × 当月天数
dspFeeSettled计提金额（结算币种）exDspFee × 汇率
rebateAmount返点金额dspFeeSettled × 返点率
returnConsume返点后消耗dspFeeSettled - rebateAmount
monthRatio月环比(本月returnConsume - 上月) / 上月
💡
关键改动：月环比基于返点后的值计算，解决当前对不上的问题。前端展示完整的数据来源链路。
// 前端展示效果 计提金额：¥14,467 返点率：10% 返点金额：-¥1,447 返点后消耗：¥13,020 月环比：+12.5%（基于 ¥13,020 与上月对比）

4.4 预计收入公式可配置化

! 现状问题

硬编码公式 exDspFee = dspFee / accrualPeriod × dayByMonth，所有业务共用。不同 DSP、不同业务场景可能需要不同的计算逻辑。

→ 优化方案：公式模板化

在 config_miads 配置表增加 formula_type 字段：

formula_type公式适用场景
DAILY_AVGdspFee / accrualPeriod × dayByMonth默认，日均推算全月
MONTHLY_RATIOdspFee × monthRatio月度比例系数
FIXEDunitPrice × volume固定单价 × 量
CUSTOM配置表达式特殊场景
// 计算引擎根据 formula_type 选择计算方法 switch (config.getFormulaType()) { case "DAILY_AVG": return AmountUtils.getExFee(dspFee, accrualPeriod, dayByMonth); case "MONTHLY_RATIO": return AmountUtils.multiply(dspFee, config.getMonthRatio()); case "FIXED": return AmountUtils.multiply(config.getUnitPrice(), volume); default: return AmountUtils.getExFee(dspFee, accrualPeriod, dayByMonth); }
